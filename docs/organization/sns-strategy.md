# Sparks Station — SNS運用方針

## 目的

Sparks Station を、AIでプロダクトを作れるようになった一方で「どう売るか」「誰がお金を払うか」に悩む日本語圏の開発者に届ける。

フォロワー数そのものより、保存・シェア・プロフィール遷移・記事閲覧・プロダクト利用への導線を重視する。

## コンテンツ方針

- 海外Micro-SaaSの成功・失敗・売却事例を短く分析する。
- 「なぜ伸びたか」「どう作ったか」に加えて、「どう課金したか」「最初の顧客をどう獲得したか」「価格をどう上げたか」まで入れる。
- サービス告知より、有益な分析コンテンツを優先する。
- 保存される「チェックリスト」「ロードマップ」「比較」「手順書」を多めにする。
- 画像内テキストでは丸数字などフォント依存文字を使わない。`1.` `2.` `3.` を使う。
- リール1枚目は「数字・金額・期間・誰でも使える方法」を入れ、抽象的な学びだけで始めない。
- リール1枚目は短くするが、短さだけを目的にしない。読者が一瞬で「何の事例か」「なぜ自分に必要か」を理解できる情報を残す。
- Threadsの1投稿目は、必要に応じて文を途中で切り、2投稿目以降を読みたくなるオープンループを作る。

## 編集テーマ

- AIで開発コストが下がった後に残る、販売・課金・継続利用の壁を扱う。
- 「作れるのに売れない」状態を突破した事例を集める。
- 価格設計、GTM、初期顧客獲得、ニッチ選定、サービス化、買収・売却、失敗理由を優先する。
- AI SaaS、Micro-SaaS、個人開発、業務代行+ソフトウェア、買収型ポートフォリオを横断して見る。
- 日本の開発者が翌日から試せる仮説、DM文、価格表、検証手順まで落とす。

## 投稿フォーマット

| 形式 | 使いどころ |
|---|---|
| Instagram Reels | フックの強い事例紹介、マネタイズ転換点のスライドショー型まとめ |
| Threadsスレッド | 事例の背景、価格設計、GTM、検証手順を連続投稿で読ませる |
| X投稿 | 記事から得た知見を、保存される短いチェックリストや日本転用アイデアとして広める |

Instagramはすべてリールに統一する。静止画・カルーセルは使わない。

ThreadsはInstagramの本文流用ではなく、別設計のスレッド投稿にする。標準は7〜9投稿。1投稿目で結論、2〜6投稿目で分解、最後に記事遷移・保存・返信募集のどれか1つだけを置く。

XはAPIを使わず、Codexのブラウザ操作で必要な時だけ投稿する。リーチ低下を避けるため、投稿本文に記事リンクは入れない。記事リンクが必要な場合は、投稿後の返信、プロフィール導線、または別途手動案内に回す。Xの基本方針は以下の2つに絞る。

- 保存されるチェックリスト型: 「作る前に見るべき4つ」「価格を決める前の3問」など、記事の学びを抽象化して持ち帰れる形にする。
- 日本転用アイデア型: 海外事例をそのまま紹介せず、日本で試すならどの業界、どの業務、どの最小実験になるかを具体化する。

### X投稿の実行フロー

市川さんが記事作成時、SNS運用時、過去記事の再利用時、または単独で「Xにも投稿して」「この記事からX投稿して」「過去記事からX投稿を作って」などと明示した場合だけ実行する。X APIは使わず、CodexのBrowser/Chrome拡張経由でログイン済みChromeを操作する。

過去記事から作る場合は、まず `data/insights/sparks-article-insights.json` を参照し、必要なら `npm run articles:analyze` で再生成する。指定された記事の `sellingPattern`、`payer`、`pain`、`japanHypothesis`、`firstExperiment`、`snsSeeds` を使い、記事本文を読み直して事実と数字を確認してから投稿文を作る。

投稿は原則2本作る。

1. チェックリスト型
   - 記事の知見を、保存される短いリストにする。
   - 例: `AIエージェント系SaaSを作る前に見るべき4つ。`
   - 構成: 問題提起 → 3〜5項目のチェックリスト → 事例からの結論。
2. 日本転用アイデア型
   - 海外事例を、日本で試せる業界・業務・最小実験へ変換する。
   - 例: `日本だとこのあたりに転用できそうです。`
   - 構成: 転用先の宣言 → 4〜6個の具体例 → 最初の売り方。

記事指定の受け取り方:

- URL指定: `/posts/[slug]` からslugを取り、記事ファイルとインサイト台帳を照合する。
- タイトル/サービス名指定: `data/insights/sparks-article-insights.json` と `src/content/posts` を検索して候補を特定する。
- テーマ指定: 価格設計、GTM、失敗事例などのテーマから、インサイト台帳の `sellingPattern` と直近SNS実績を見て候補記事を選ぶ。

承認から投稿までの流れ:

1. Codexが候補記事を選び、X投稿案を2通り提示する。
   - チェックリスト型
   - 日本転用アイデア型
2. 市川さんが「投稿して」「これで」「OK」「お願いします」など承認したら、その承認を実投稿の明示指示として扱う。
3. CodexはBrowser/Chrome拡張でログイン済みChromeのXを開く。
4. 1本目を投稿欄へ入力する。
5. 投稿前に以下を確認する。
   - 本文にURLがない
   - ドメイン風表記が自動カード化されていない
   - 意図しない画像、カード、引用が付いていない
6. 問題なければ公開する。
7. タイムラインまたはプロフィール上で投稿本文が表示されたことを確認し、投稿URLを控える。
8. 2本目も同じ手順で投稿する。
9. 最後に2本分の投稿URLを市川さんへ返す。

実行時の注意:

- 本文に記事URLや外部URLを入れない。
- `Meerkats.ai` のようなドメイン風表記は自動カード化されるため、必要に応じて `Meerkats AI` のように表記を変える。
- 投稿前にXのDOMまたは画面で、自動カードプレビューが出ていないことを確認する。
- 公開ボタンを押した後、投稿本文がタイムライン上に表示されたことを確認し、投稿URLを控える。
- パスワードや2FAコードはCodexが扱わない。ログインが必要な場合は市川さんにChrome側で対応してもらう。

リールは動画プロンプトから直接生成しない。キューに含めた `instagram.slides[].imagePrompt` を使って各スライド画像を生成し、その画像に `overlayText` を合成してから、ffmpegでスライドショー形式のMP4に変換する。

## 投稿比率

- 50%: 海外事例から見るマネタイズ突破パターン
- 25%: AI時代のGTM、価格設計、初期顧客獲得
- 15%: 失敗事例と「売れなかった理由」
- 10%: Sparks Stationの記事・プロダクト導線

## 週次運用

投稿は毎日その場で自動生成しない。市川さんが「次の週のSNS投稿案を作成して」など、SNS投稿案作成を明示した時だけ、レオがテーマと優先順位を決め、ルカがInstagram ReelsとThreadsスレッドを1週間分まとめて作成し、Firestore `postsQueue` へ投入する。

投稿案生成はCodexとの対面作業で行う。APIは投稿案を生成しない。APIが行うのは、Firestoreキューの取得、`instagram.slides[].imagePrompt` に基づくリール用画像生成、画像へのテキスト合成、MP4化、Instagram/Threads投稿のみ。

市川さんからSNS投稿案作成の明示指示がない場合、Codexはインサイト台帳や候補JSONの更新までは行ってよいが、週次投稿案の本文作成やFirestore投入は開始しない。

### 伸びた投稿の再設計

投稿ペースを維持するため、毎回すべてを新ネタにしようとしない。インサイトで伸びた投稿は、同じ記事・同じ事例を別視点で作り直し、再投稿候補として扱う。

再投稿は単純な再掲ではなく、以下のどれかの切り口へ変換する。

- 事例紹介: 何が起きたか、どの数字が重要か。
- 価格設計: なぜ払われたか、どの料金モデルが効いたか。
- GTM: 最初の顧客をどこで獲得したか。
- 失敗診断: どこで詰まる可能性があったか。
- 日本転用: 日本の個人開発者ならどう試すか。
- チェックリスト: 作る前に確認する項目へ落とす。
- 比較: 似た事例との差分を見せる。

週次分析では、ルカが必ず「再投稿候補」を出す。候補には、元投稿、伸びた理由の仮説、新しい切り口、新しいフック、CTA変更案を含める。

再投稿候補の優先順位は、保存・シェア・Threadsの閲覧/返信/引用、記事遷移の順で判断する。特に保存またはThreads閲覧が強い投稿は、翌週以降に視点を変えて再利用する。

同じ記事を使う場合でも、InstagramとThreadsで役割を変える。Instagramは保存される短い診断・チェックリスト型、Threadsは背景・理由・日本転用まで読ませる分解型を優先する。

基本フロー:

1. `npm run sns:insights` で最新のInstagram / Threadsインサイトを取得する。
2. `npm run articles:analyze` で記事インサイト台帳とSNS候補JSONを更新する。
3. レオが直近投稿の数字、テーマ、フック、CTA、Threadsの離脱を確認する。
4. ルカがインサイトを分析し、現行方針に対する改善候補と再投稿候補を出す。
5. 方針変更が必要な場合は、市川さんの許可を得てから `docs/organization/sns-strategy.md` に反映する。
6. レオが今週のテーマを決める。
7. ルカが7日分の投稿案を作る。
8. レオが事例、数字、出典、CTAを確認する。
9. 承認済み候補を `approved: true` にし、投稿本文、`hookText`、`slides`、`threadPosts`、`scheduledAt` を埋める。
10. `npm run sns:seed-approved -- --file=tmp/sns-approved-candidates.json` でFirestoreへ投入する。
11. `node scripts/check-sns-queue.mjs` で `pending` 件数と投稿順を確認する。

週次作成時の目安:

| 曜日 | テーマ | 目的 |
|---|---|---|
| 月 | 成功事例のマネタイズ分解 | 保存・プロフィール遷移 |
| 火 | 価格設計 / 値上げ | 保存・シェア |
| 水 | 初期顧客獲得 / GTM | 返信・記事遷移 |
| 木 | 失敗事例 / 売れなかった理由 | 信頼形成・保存 |
| 金 | 買収 / 売却 / B2B化 | 高単価テーマの反応確認 |
| 土 | 実践チェックリスト | 保存・再訪 |
| 日 | 週次まとめ / 記事導線 | 記事閲覧・次週への接続 |

キュー投入は `scheduledAt` を持たせ、Cronは `scheduledAt <= now` かつ `status == "pending"` の投稿を処理する形に拡張するのが理想。初期実装では `order` 昇順で処理し、週次投入時に投稿順を管理する。

| 買収/売却の読み解き | 「この小さなSaaSが買われた本当の理由」 | 20〜28秒 / 5〜6枚 | 売却/買収額 / 買い手の狙い / 顧客資産 / 技術資産 / 再現できる要素 / 記事CTA | 買収目線で保存 | ニッチSaaS売却、買収型ポートフォリオ |

画像内テキストは1枚あたり最大2行、1行12〜14字を目安にする。必ず「$49」「月$12k」「3つ」「7日」「B2B」「解約率」など、数字・価格・課金理由を残す。

ただし、圧縮しすぎて「何のことかわからない」フックにしない。1枚目は、短い中にも以下のどれか2つ以上を入れる。

- 事例カテゴリ: AI SaaS、フォームSaaS、買収、価格設計、失敗事例など
- 読者の痛み: 売れない、価格が決められない、初期顧客が取れない、解約が多いなど
- 数字: MRR、売却額、期間、顧客数、先行販売額など
- 学びの方向: 課金理由、手動DM、価格不満、無料枠、規制、支払い手など

例:

- 良い: `Typeform不満を拾い\n月$18k`
- 良い: `作る前に300万円\n売ったAI SaaS`
- 弱い: `課金理由はここ`
- 弱い: `価格を分解`

1枚目のフックは、次の型を優先する。

- `誰でも使える [金額/成果] の型`
- `[期間]で[成果]した [会社/個人] の方法`
- `[人数/体制]で[売上] まで伸ばした理由`
- `[調達額/技術力] があっても失敗した理由`
- `作る前に [金額] 売った [カテゴリ]`

ただし、煽りすぎず、記事で確認できる事実に基づく表現にする。

| テンプレート | 1投稿目 | 2投稿目以降 | CTA | 向いている事例 |

| 価格設計スレッド | 「価格を決める時、“安ければ売れる”はだいたい罠です。」 | 安い価格の弊害 / 払う顧客の条件 / プランの分け方 / 値上げのタイミング / 事例 / 自分ならこう試す | 料金表を作る前に読み返す | 課金、値上げ、B2Bプラン、サブスク |

| 市場選定スレッド | 「AIで作れる時代ほど、“何を作らないか”が大事です。」 | 伸びる市場の条件 / 避けたい市場 / 支払い能力 / 頻度と深刻度 / 事例 / チェックリスト | 市場選定メモとして保存 | 新規テーマ探索、ニッチ選定、記事導線 |

Threadsでは1投稿目の完成度を最優先する。インサイト上、1投稿目に閲覧が集中しやすいため、1投稿目だけで「結論」「対象読者」「続きを読む理由」が伝わるようにする。

InstagramとThreadsのCTAは役割を分ける。

- Instagram: 保存、プロフィール遷移を主目的にする。
- Threads: 記事流入、返信、引用/リポストを主目的にする。
- 1つの投稿セットで同じCTAを重ねすぎず、媒体ごとの役割を明確にする。

## Firestoreキュー

コレクション名: `postsQueue`

```json
{
  "platforms": ["instagram", "threads"],
  "contentTheme": "pricing",
  "caseName": "Example SaaS",
  "sourceUrl": "https://...",
  "articleUrl": "https://sparks-station.com/...",
  "type": "reel",
  "order": 1,
  "status": "pending",
  "scheduledAt": "timestamp",
  "instagram": {
    "type": "reel",
    "templateId": "reel_pricing_breakdown",
    "hookText": "月$12kの理由",
    "caption": "Instagram本文",
    "slides": [
      {
        "role": "hook",
        "imagePrompt": "English image prompt for the generated reel slide image, vertical 9:16, no text, no letters",
        "overlayText": "月$12k\n課金理由は1つ"
      }
    ],
    "ctaType": "save",
    "targetKpi": "saves"
  },
  "threads": {
    "templateId": "thread_case_breakdown",
    "posts": [
      "1投稿目の本文",
      "2投稿目の本文",
      "3投稿目の本文"
    ],
    "ctaType": "article_click",
    "targetKpi": "link_clicks"
  },
  "tracking": {
    "utmCampaign": "sns_2026_w01",
    "experimentId": "hook_price_001"
  },
  "result": {
    "instagramPostId": null,
    "threadsRootId": null,
    "threadsPostIds": [],
    "postedAt": null
  },
  "createdAt": "server/client timestamp"
}
```

`instagram.slides[].imagePrompt` は必須。ここにリール用画像の生成プロンプトを入れる。プロンプトは英語で書き、`vertical 9:16`, `no text`, `no letters` を含める。文字情報は画像生成に含めず、`overlayText` として後から合成する。

## 自動投稿API

- Route: `POST /api/instagram/post`
- Cron 実行時: `x-cron-secret` ヘッダーに `CRON_SECRET` を渡す。
- 手動投稿時: Firebase Admin 権限の ID token を `Authorization: Bearer <token>` で渡す。

必要な環境変数:

```text
NEXT_PUBLIC_FIREBASE_PROJECT_ID=
FIREBASE_ADMIN_CLIENT_EMAIL=
FIREBASE_ADMIN_PRIVATE_KEY=
ADMIN_UIDS=
GEMINI_API_KEY=
CLOUDINARY_CLOUD_NAME=
CLOUDINARY_API_KEY=
CLOUDINARY_API_SECRET=
INSTAGRAM_BUSINESS_ACCOUNT_ID=
INSTAGRAM_ACCESS_TOKEN=
THREADS_USER_ID=
THREADS_ACCESS_TOKEN=
CRON_SECRET=
REEL_BGM_URL=
```
